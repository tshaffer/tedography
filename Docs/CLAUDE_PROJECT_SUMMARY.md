# Tedography — Project Summary

> Generated by Claude after examining the full codebase and documentation.

---

## What It Is

**Tedography** is a personal photo archive and curation system — a redesign of an earlier project called *Shafferography*. It is a **pnpm monorepo** written entirely in TypeScript, structured as a full-stack web application for a single photographer managing a local archive.

The primary workflow is:

```
Import → Review → Keep / Pending / Discard → Organize → Browse
```

---

## Architecture

### Monorepo Structure

| Path | Purpose |
|---|---|
| `apps/web` | React 19 + Redux Toolkit + Material UI frontend (Webpack) |
| `apps/api` | Express 5 backend (runs via `tsx`) |
| `apps/duplicate-cli` | CLI tool for duplicate detection |
| `apps/duplicate-review-web` | Web UI for reviewing duplicates |
| `apps/duplicate-worker` | Background worker for duplicate processing |
| `packages/domain` | Canonical data model — entities, enums, types |
| `packages/shared` | Utilities shared across apps and packages |
| `packages/media-metadata` | EXIF extraction, image dimensions, MIME types |
| `packages/import-pipeline` | File hashing, duplicate detection, asset creation |
| `packages/duplicate-domain` | Types for duplicate detection workflows |
| `packages/image-analysis` | Image analysis support |

### Dependency Direction

```
domain
   ↑
shared
   ↑
media-metadata
   ↑
import-pipeline
   ↑
api

web → domain, shared
```

No circular dependencies. `domain` must stay dependency-light.

### Persistence

MongoDB via Mongoose. Key collections:

- `mediaAssets`
- `albumTreeNodes`
- `people`
- `keywords`
- `smartAlbums`
- `faceDetections`
- `faceMatchReviews`
- `personFaceExamples`

### Media Storage

Photo and video files live on **local drives** — not in MongoDB. MongoDB stores metadata only. The API serves media via three routes:

- `GET /media/display/:assetId`
- `GET /media/thumbnail/:assetId`
- `GET /media/original/:assetId`

## Core Domain Model

| Entity | Purpose |
|---|---|
| `MediaAsset` | A photo or video — the central entity |
| `AlbumTreeNode` | Hierarchical album tree (Groups + Albums) |
| `Person` | A person who appears in photos |
| `Keyword` | Hierarchical keyword/tag |
| `SmartAlbum` | Saved filter preset (by keyword, state, year) |
| `FaceDetection` | Result of face detection on an asset |
| `FaceMatchReview` | Pending human review of a face match |
| `PersonFaceExample` | Reference face crop for a known person |
| `PendingGroup` | Named group of photos awaiting decision |
| `StorageInstance` | A copy of an asset at a storage location |

### PhotoState Lifecycle

```
New → Pending → Keep → Discard
```

- **New** — freshly imported, not yet reviewed
- **Pending** — deferred for later decision (can be grouped into named Pending Groups)
- **Keep** — confirmed keeper
- **Discard** — rejected (hidden by default but recoverable)

## Key Workflows

### Import Pipeline (`registerImportService.ts`)

1. Browse available storage roots
2. Scan a target folder to enumerate media files
3. Select files and register them

For each file during registration:
1. Hash the file (SHA-256)
2. Check for duplicates by path and hash
3. Extract EXIF metadata (capture date/time, dimensions, camera info, GPS/location with reverse geocoding)
4. Convert HEIC → JPEG if needed (stored in derived root)
5. Generate a JPEG thumbnail (stored in derived root)
6. Create a `MediaAsset` record in MongoDB
7. Schedule the people pipeline for the new asset

### People Pipeline

Optional AWS Rekognition-based face detection and matching. Configurable via environment variables. Supports `rekognition`, `mock`, and `none` engine modes. Produces `FaceDetection` records, which flow through `FaceMatchReview` for human confirmation before being attached to `Person` entities on `MediaAsset.people`.

### Review UI

The main `App.tsx` is a large single-component UI managing:

- Grid, Loupe, Survey, and Filmstrip viewing modes
- Photo state transitions (Keep / Discard / Pending)
- Album management (create, rename, move, reorder)
- Keyword tagging
- Search and filter (by state, album, people, date range, filename pattern, publication status)
- People review dialogs
- Smart albums

### Album Tree

Hierarchical Groups + Albums. Assets belong **only to leaf albums** — parent nodes are organizational only. Supports:

- Manual drag-and-drop ordering within albums
- Child order modes: `Custom`, `Name`, `NumericThenName`
- Move assets between albums
- Move album nodes within the tree

### Smart Albums

Saved filter presets that combine: keyword, photo state, and/or year group. Listed in the sidebar for quick access.

### Maintenance Tools

Available from the Maintenance dialog and via CLI scripts:

- Thumbnail verify / repair / rebuild
- Asset integrity verification
- Location metadata backfill
- Reimport known assets in a folder
- Rebuild derived files (display + thumbnail) for a folder

## Running the Project

```bash
# Install dependencies
pnpm install

# Start both web and API together
pnpm dev

# Start individually
pnpm dev:web
pnpm dev:api

# Typecheck all packages
pnpm -r typecheck

# Thumbnail maintenance
pnpm thumbnails:verify
pnpm thumbnails:repair
pnpm thumbnails:rebuild
```

### Environment Configuration

The API reads from `apps/api/.env`. Key variables:

| Variable | Description |
|---|---|
| `MONGODB_URI` | MongoDB connection string |
| `TEDOGRAPHY_STORAGE_ROOTS` | Semicolon-separated list of `id\|label\|/absolute/path` |
| `TEDOGRAPHY_DERIVED_ROOT` | Absolute path for derived files (converted HEIC, thumbnails) |
| `TEDOGRAPHY_UNROTATED_ROOT` | Optional path for pre-rotation originals |
| `TEDOGRAPHY_PEOPLE_PIPELINE_ENABLED` | `true` / `false` |
| `TEDOGRAPHY_PEOPLE_PIPELINE_ENGINE` | `rekognition` / `mock` / `none` |

## Rules for Making Changes

1. **`.js` extensions on all relative imports** — the project uses `NodeNext` module resolution. Do not remove them.
2. **Respect dependency direction** — see diagram above. No circular deps.
3. **Don't rename `PhotoState` values** without updating domain enums, API logic, and UI together.
4. **Keep `domain` dependency-light** — no internal package dependencies inside `domain`.
5. **Run `pnpm -r typecheck` after every change** to verify correctness across all packages.
6. **Prefer small, focused patches** — avoid unnecessary refactors.
7. **Relative imports within a package are fine** — but prefer workspace imports (`@tedography/domain`) across packages.
